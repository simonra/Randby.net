About

The data hub team worked with establishing a data hub for the organization. The goal was to have a platform that enabled the product teams to have a reliable, yet easy to use, distributed data store, that should scale well as the amount of data grows, as well as facilitating collaboration with other teams about the data. The project evaluated different technologies, proof of concept-ed the platform from installation to operations to usage management, procured licenses for operations support and tooling for the development teams, as well as handing over the established infrastructure to the operations team. It also involved working with other development teams on the adaptation, as well as the product and business management to work out how to best utilize the platform we were establishing to deliver more value at less cost. We also had extensive dialog with the legal department on how we could enable more secure treatment of data.

Technology

The core work was setting up Kafka and securing it with mTLS, as well as creating a small management tool in C# to manage it, which we ran in Kubernetes. We found Kafka to be a very forgiving piece of infrastructure to operate at a small scale, so we had time to do a good job with the public/government procurement process that would enable us to have better tooling than what we'd have time to maintain in house, as well as proper support for the operations teams when they'd take over the operations. The final tooling gave us an opportunity to explore integrations between Blazor, Keycloak, and Apicurio. We also worked a lot with creating creating usage examples for various scenarios, both in dotnet and Java which many of the teams used, but also hosting and running Kafka Connectors. Finally we also set up OpenTelemetry integrations directly with Kafka, and worked on increasing the adaptation of OpenTelemetry among the development teams.

Role

Even though I was a developer, and produced a fair bit of code both for tooling and examples/starting points for the development teams, the high impact from the work came from the dialog with the rest of the organization.

We worked on automating all of it with Terraform and the Azure Resource Manager templates.

As the project went on, I took on the responsibilities of the Quality Champion, then also the Security Architect, and finally the Security Champion as they moved on to other projects.

About

The Grunndata project was the project that gathered and distributed the core data which the Norwegian Health Care Sector used to interoperate.

Technology

The main product tech stack was a Microsoft SQL Server, with a C# backend, that published mostly WCF (Microsoft SOAP) integration endpoints, but also a substantial amount of REST endpoints. The solution had a diverse set of front end technologies, spanning .NET MVC, jQuery, AngularJS and Angular. Parts of the front end were managed through Microsofts IIS web server. It was also tightly integrated with a Microsoft ServiceBus, and it all came together through a deep intertwinement with Windows Server and Active Directory.

Role

As the Tech Lead I was still mostly focused on development. However, my role also entailed working out the way forward for the product together with owners at the department of health. Further, I maintained the Development and Architecture Departments CI/CD servers and services (TFS), device management (AD), VPN (pfSense), dev network (haproxy), dev/test server infrastructure (mostly Windows Server and SQL Server), and managing the organizations presence on GitHub for public API contracts. I got to participate in most of the more arcane debugging (occasional memory dumps from crashes in production from the Operations department, self patching data proxies installed at customers), as well as contributing to a large portion of the code reviews.
